4. Dashboard falla WCAG 2.2 AA (0 aria-, 0 alt, 0 role).
5. Sinapsis bridge no auto-detecta `norteia-continuous-learning` (directorio real del operador) — solo `sinapsis-learning`.

### 2.2 Performance Efficiency — **2.3 / 5**

| Subcarac. | Score | Evidencia |
|---|---|---|
| Time Behaviour | **2** | **435 ms/hook medidos en Windows Git Bash** (p100 = 465 ms). Target del prompt = `< 200 ms`. Phase-detector + mode-injector juntos = ~**900 ms añadidos por turno** antes de que Claude empiece a responder. Gate-validator añade ~435 ms a cada Write/Edit. Baseline cold-start bash+python3 ya consume ~200 ms, así que el hook mismo solo puede bajar a ~230 ms si se elimina toda lógica. |
| Resource Util. | 3 | Python stdlib-only, sin deps. Stdin cap 1 MiB (verificado). Memoria estimada por hook ~15 MB (intérprete Python frío). |
| Capacity | 2 | Logs `gate-validator.log` etc. crecen sin rotación hasta `session-closer` los parte al cerrar. Entre sesiones largas sin cierre, 1 proyecto activo + 100 gates/turno × 50 turnos = 5 000 líneas en memoria cuando session-closer arranque. No probado con 10 000 líneas. SKILL.md de 50 KB: el smart-truncate de `mode-injector:84` corta por `---` / `## ` con presupuesto 6 KB/modo, 16 KB total → tolera SKILL.md grandes. |

**Qué moverá la aguja a 3.5 en v1.2:**
- **Modo cacheado**: leer `_operator-config.json` una sola vez al arrancar `mode-injector` y no re-abrir SKILL.md de modos desactivados.
- **Activación condicional**: saltar `phase-detector` si el prompt no contiene ninguna palabra de `_passive-triggers.json → phaseDetection` (early-exit en bash antes de arrancar python3; ahorra ~200 ms).
- **Benchmark automatizado** en CI (`tests/perf/`): guard `<300 ms por hook en Linux`.

**Qué moverá la aguja a 4.5 en v2.0:**
- Migrar hooks a un único worker Python persistente (daemon) que reciba JSON por socket Unix o named pipe. Reduce cold-start a ~5 ms/hook.
